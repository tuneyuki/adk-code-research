# ADK Runner について: 目的と必要性

## 1. Runner の目的と機能

**`Runner`** は、単なる「実行役」以上の、**Agent アプリケーションのランタイム環境そのもの**を提供するクラスです。Agent 単体では持っていない「会話の状態管理」や「システム連携」を一手に引き受けます。

主な機能は以下の通りです：

*   **セッション管理 (Session Management)**:
    *   ユーザーID、セッションIDに紐づく会話履歴（イベント）の保存と読み込みを行います。
    *   `SessionService` (例: `InMemorySessionService`, Firestoreなど) と連携し、会話の状態を永続化・復元します。
*   **コンテキストの構築 (Context Setup)**:
    *   Agent が動作するために必要な `InvocationContext` を準備します。これにはプラグイン、設定、履歴などが含まれます。
*   **イベント処理 (Event Handling)**:
    *   ユーザーからのメッセージを受け取り、Agent に渡し、Agent が生成したイベント（回答、ツール呼び出しなど）をセッション履歴に追記 (`append_event`) します。
    *   **ここが重要**: Agent 自体は「入力を受けて出力を返すロジック」であり、その出力を「履歴として保存する」機能は Runner が担っています。

## 2. Runner を使わずに公開できるか？

**結論: 原則として Runner (または同等の仕組み) は必須です。**

技術的には `Agent.run_async()` メソッドを直接呼び出すことは可能ですが、以下の理由から現実的ではありません：

1.  **記憶喪失になる**: Runner を使わないと、`SessionService` との連携が自動で行われません。つまり、直前の会話内容を記憶したり、会話履歴を保存したりする仕組みを**すべて自分で実装する**必要があります。
2.  **複雑なセットアップが必要**: `Agent.run_async()` は引数として `InvocationContext` を要求します。これを正しく構築するには、手動でプラグインマネージャーや設定オブジェクトを生成する必要があり、Runner の内部ロジックを再発明することになります。

### まとめ

*   **Runner は「OS」、Agent は「アプリ」**: Agent はロジックですが、それが動くためにはメモリ管理や入出力管理を行う OS (Runner) が必要です。
*   **サービス公開には必須**: ステートフルな（文脈を維持した）チャットボットとして公開する場合、Runner が提供するセッション管理機能が不可欠です。

独自のランタイムを構築したいという強い動機がない限り、標準の `Runner` を使用することを強く推奨します。
