# Agentのセッション永続性と再起動時の挙動

このドキュメントでは、`InMemorySessionService` を使用している環境において、サーバー（またはアプリケーション）が再起動した際のAgentおよびセッションの挙動について解説します。

## 概要

`main.py` でデフォルトで使用されている `InMemorySessionService` は、その名の通り**メモリ上（RAM）**にのみセッションデータを保持します。

したがって、以下の挙動となります：

1.  **データの揮発性**: サーバープロセスが終了すると、全てのセッション履歴（会話ログ、コンテキスト）は完全に失われます。
2.  **再起動後のID無効化**: サーバー再起動後に、再起動前に発行された `session_id` を使用してリクエストを送っても、サーバー側はそのIDを認識できません。

## 詳細な挙動フロー

### 1. 正常なフロー（再起動前）
1.  クライアントが `run_agent` 等を開始。
2.  サーバーが `session_id="session_A"` でセッションを作成。
3.  会話履歴がメモリ上の辞書（Dictionary）に保存される。
4.  会話が継続可能。

### 2. サーバー再起動
*   プロセスが終了した時点で、メモリ上の `session_A` のデータは消滅します。
*   再起動時、`InMemorySessionService` は空の状態（セッション数 0）で初期化されます。

### 3. 再起動後のリクエスト（エラー発生）
クライアント側が `session_id="session_A"` を保持しており、再起動後のサーバーに対してこのIDでリクエストを送信した場合：

*   **動作**: `Runner` が `session_service.get_session("session_A")` を呼び出します。
*   **結果**: `InMemorySessionService` はこのIDを知らないため、`None` (または該当なし) となります。
*   **エラー**: ADKの仕様上、存在しない `session_id` を指定して `run` を実行しようとすると、**`ValueError: Session not found: session_A`** が発生します。

**注意点**:
自動的に新しいセッションとして作り直される（暗黙の再生成）ことは**ありません**。クライアントは明示的に新しいセッションを作成するか、サーバー側でエラーハンドリングを行う必要があります。

## 永続化が必要な場合

本番環境や、再起動後も会話を継続したい場合は、`InMemorySessionService` ではなく、永続化に対応したSessionServiceの実装を使用する必要があります（例: Firestore, Redis, SQL Database等をバックエンドとする実装）。

現状の `google-adk` パッケージに含まれる標準実装を確認し、適切な永続化バックエンドを選択してください。
